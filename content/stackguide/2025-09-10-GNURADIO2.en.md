## GNU Radio Companion Interface

### Launch and first impression

```bash
# Launch GNU Radio Companion
gnuradio-companion

# Or from Applications menu (depending on distribution)
# Applications → Engineering → GNU Radio Companion
```

GRC's interface consists of four main areas:

1. **Toolbar**: New, Open, Save, Execute, Stop
2. **Block palette** (left): Catalog of available blocks, organized by category
3. **Central canvas**: Workspace for assembling the flowgraph
4. **Console** (bottom): Error messages, warnings, and program output

### Block palette organization

Blocks are organized in logical categories:

- **Core**: Fundamental blocks (sources, sinks, operators)
- **Filters**: FIR, IIR filters, decimators, interpolators  
- **Math Operations**: Mathematical operations, FFT, correlation
- **Modulators/Demodulators**: AM, FM, PSK, QAM, etc.
- **GUI**: Graphical interfaces (scopes, spectra, controls)
- **Audio**: System audio sources and sinks
- **Blocks**: Generic and utility blocks

**Practical test**: Count the number of blocks available in your installation. A complete GNU Radio 3.10+ installation contains over 200 basic blocks.

## Fundamental concepts: blocks and connections

### Blocks: basic units

Each block in GNU Radio represents a signal processing function:

**Sources**: Generate data
- `Signal Source`: Generates sinusoids, squares, triangles
- `Noise Source`: White, Gaussian noise
- `File Source`: Reads data files

**Sinks**: Consume data
- `QT GUI Time Sink`: Time-domain oscilloscope
- `QT GUI Frequency Sink`: Spectrum analyzer
- `File Sink`: Writes to file

**Processing blocks**: Transform data
- `Multiply`: Signal multiplication
- `Add`: Signal addition
- `Low Pass Filter`: Low-pass filter

### Connection types

GNU Radio uses a strict type system:

- **Complex** (blue): Complex I/Q data
- **Float** (orange): Real floating-point numbers
- **Int** (yellow): Integer numbers
- **Byte** (green): 8-bit data

**Important rule**: You can only connect ports of the same type. GRC indicates type errors with red connections.

## First simple flowgraph

### Basic signal generator

Let's create a first flowgraph to generate and visualize a signal:

1. **Adding blocks**:
   - Drag `Signal Source` from Core → Sources
   - Add `Throttle` from Core → Misc  
   - Add `QT GUI Time Sink` from Core → GUI

2. **Parameter configuration**:

   **Signal Source**:
   - `Output Type`: Complex
   - `Sample Rate`: 32000
   - `Waveform`: Cosine
   - `Frequency`: 1000
   - `Amplitude`: 1

   **Throttle**:
   - `Type`: Complex
   - `Sample Rate`: 32000

   **QT GUI Time Sink**:
   - `Type`: Complex
   - `Sample Rate`: 32000
   - `Number of Points`: 1024

3. **Connections**:
   ```
   Signal Source → Throttle → QT GUI Time Sink
   ```

### Flowgraph analysis

This simple flowgraph illustrates key concepts:

- **Signal Source** generates a complex sinusoid at 1 kHz
- **Throttle** limits throughput to avoid CPU overload
- **Time Sink** displays the signal in real time

**Why the Throttle?** Without it, GNU Radio would generate data as fast as possible, saturating the CPU. The Throttle simulates a realistic sampling frequency.

## Data types and sampling frequencies

### Type correspondence

| GRC Type | Description | Size | Typical usage |
|----------|-------------|------|---------------|
| complex | Complex I/Q | 8 bytes | RF signals, baseband |
| float | 32-bit float | 4 bytes | Audio, real signals |
| int | 32-bit integer | 4 bytes | Counters, control |
| byte | 8-bit integer | 1 byte | Binary data |

### Sampling frequency management

**Golden rule**: All blocks in a path must have the same sampling frequency, except when explicitly using decimators/interpolators.

**Throughput calculation example**:
- Complex signal at 2 MSps = 2×10⁶ × 8 bytes = 16 MB/s
- Float signal at 48 kSps = 48×10³ × 4 bytes = 192 kB/s

Create an intentional error:

## Execution and debugging

### Execution modes

GRC offers several execution modes:

**Normal mode** (F5):
- Generates Python code
- Compiles if necessary  
- Launches execution

**Step-by-step mode**:
- `Generate` (F7): Generates code only
- `Execute` (F6): Launches generated code

### Common error messages

**1. Connection error**:
```
RuntimeError: cannot connect output port X to input port Y
```
*Solution*: Check data types (connection colors)

**2. Missing parameter**:
```
Missing required parameter: Sample Rate
```
*Solution*: Fill all mandatory fields (in red)

**3. Unconnected flowgraph**:
```
Flow graph has unconnected blocks
```
*Solution*: All blocks must be connected in a coherent path

### Debug console

The console displays useful information:
- **INFO**: General execution information
- **WARNING**: Non-blocking warnings  
- **ERROR**: Errors preventing execution
- **DEBUG**: Detailed information (verbose mode)

## Python code generation

### Automatically generated code

GRC automatically generates equivalent Python code. For a simple flowgraph:

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-

from gnuradio import blocks
from gnuradio import gr
from gnuradio import qtgui
from gnuradio.filter import firdes

class simple_flowgraph(gr.top_block):
    def __init__(self):
        gr.top_block.__init__(self, "Simple Signal Generator")
        
        # Blocks
        self.signal_source = blocks.signal_source_c(
            32000,          # Sample rate
            blocks.GR_COS_WAVE,  # Waveform
            1000,           # Frequency
            1,              # Amplitude
            0               # Offset
        )
        
        self.throttle = blocks.throttle(
            gr.sizeof_gr_complex*1, 
            32000,
            True
        )
        
        self.qtgui_time_sink = qtgui.time_sink_c(
            1024,           # Size
            32000,          # Sample rate
            "",             # Name
            1               # Number of inputs
        )
        
        # Connections
        self.connect((self.signal_source, 0), (self.throttle, 0))
        self.connect((self.throttle, 0), (self.qtgui_time_sink, 0))
```

**For learning**:

**For prototyping**:
- Rapid parameter modification

## Comparison with other visual tools

### GNU Radio Companion vs alternatives

| Tool | Type | Advantages | Disadvantages |
|------|------|------------|---------------|
| **GRC** | Open source | Free, extensible, active community | Learning curve |
| **MATLAB Simulink** | Proprietary | Polished interface, MATLAB integration | Cost, lock-in |
| **LabVIEW** | Proprietary | Excellent for instrumentation | Very expensive, specialized |
| **SDR# Plugins** | Free/limited | Easy for reception | Limited, Windows only |

### GRC specific strengths

1. **Complete ecosystem**: Over 200 basic blocks
2. **Open source**: Possibility to modify/extend
3. **Multi-platform**: Linux, Windows, macOS
4. **Code generation**: Easy transition to pure Python
5. **Active community**: Third-party blocks, examples, support

### Optimal use cases

**GRC excels for**:
- Rapid prototyping of SDR algorithms
- Education and training
- Concept testing before implementation
- Simple/medium real-time applications

**GRC limitations**:
- Limited performance for critical applications
- Sometimes slow interface with complex flowgraphs
- Difficult debugging on large projects
